1. Introduction

This memo defines how to use network management protocols in the My Teams project.

2. Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14, RFC 2119 [RFC2119].

3. Overview

4. Client

4.1. Overview

The client must be able to connect to a server, to send multiple commands to it, to receive replies from it and to display the information to the user.

The client must send command in a specific format described later and should display the reply in a human readable format.

You can find commands and reply-codes in Commands Section(6) and Reply-Codes Section(7)

4.2. Send

The client must send commands to server in a particular format. Each command must be sent with a specific header as follows: 

          
                   
typedef struct __attribute__((packed))
{
    unsigned char command;
    int message_size;
} client_request_t;

command: An unsigned char from this list corresponding to the command associated with the request. Allowed commands are: 

          
                   
enum command_code_e {
    UNKNOWN,
    HELP,
    LOGIN,
    LOGOUT,
    USERS,
    USER,
    SEND,
    MESSAGES,
    SUBSCRIBE,
    SUBSCRIBED,
    UNSUBSCRIBE,
    USE,
    CREATE,
    LIST,
    INFO
};

message_size: The total size in bytes of all arguments of the associated command after the header.

  • The header has a fixed size (5 bytes).
  • The client should verify the number and type of each argument of the associated command. There is another verification server side.
  • The header is followed by the command's arguments.

4.3. Receive

The client receives replies from the server in a particular format. Each reply is preceded by a request header in a specific format as shown below: 

          
                   typedef struct __attribute__((packed))
                   {
                   unsigned short reply;
                   int message_size;
                   } server_reply_t;

reply: An unsigned short from this list corresponding to the reply-code associated with the reply. Allowed reply codes are: 

                             
enum reply_code_e {
    COMMAND_OK = 200,
    PRINT_USER = 201,
    PRINT_USERS = 202,
    PRINT_TEAM = 210,
    PRINT_CHANNEL = 211,
    PRINT_THREAD = 212,
    PRINT_PRIVATE_MESSAGES = 220,
    PRINT_TEAMS = 230,
    PRINT_CHANNELS = 231,
    PRINT_THREADS = 232,
    PRINT_REPLIES = 233,
    PRINT_SUBSCRIBED = 240,
    PRINT_UNSUBSCRIBED = 241,
    PRINT_TEAM_CREATED = 250,
    PRINT_CHANNEL_CREATED = 251,
    PRINT_THREAD_CREATED = 252,
    PRINT_REPLY_CREATED = 253,
    
    EVENT_TEAM_CREATED = 311,
    EVENT_CHANNEL_CREATED = 312,
    EVENT_THREAD_CREATED = 313,
    EVENT_THREAD_MESSAGE_RECEIVED = 320,
    EVENT_PRIVATE_MESSAGE_RECEIVED = 321,
    EVENT_LOGGED_IN = 330,
    EVENT_LOGGED_OUT = 331,
    
    SYNTAX_ERROR = 500,
    SYNTAX_ERROR_ARGS = 501,
    NOT_IMPLEMENTED = 502,
    BAD_SEQUENCE = 503,
    NOT_IMPLEMENTED_ARG = 504,
    NOT_LOGGED_IN = 530,
    CLIENT_ALREADY_EXISTS = 540,
    UNKNOWN_COMMAND = 550,
    UNKNOWN_TEAM = 551,
    UNKNOWN_CHANNEL = 552,
    UNKNOWN_THREAD = 553,
    UNKNOWN_USER = 554,
    UNAUTHORIZED = 560,
};

message_size: The total size in bytes of all arguments of the reply after the header.

  • The header has a fixed size (6 bytes).
  • The header is followed by the arguments.

5. Server

5.1. Overview

The server must be able to accept multiple clients, to send replies to a specific client, and to broadcast a message to multiple clients.

The server must be able to keep data about users, teams, channels and threads. Clients with the right permissions must be able to access that data. The server should store the data so it can be reused after a restart.

Users must be able to create teams, channels, and threads. They also must be able to create replies in threads and send private messages to other users. All of this information must be able to be accessed by users if they have the right permissions.

Some events must be broadcasted to multiple connected clients, please refer to the Reply Codes and Commands sections for more information about broadcasted events.

5.2. Send

The server must answer clients' requests with replies. A reply starts with a header describing the reply (unsigned short: fisrt two bytes) and the size of the arguments (int: last four bytes). The header is followed by the arguments of the reply.

You may use the following structure for server replies: 

          
                  
 typedef struct __attribute__((packed))
 {
     unsigned short reply;
     int message_size;
 } server_reply_t;

reply represents the reply code. Please refer to the Reply Code section for more information.

messaege_size represents the size of the arguments. Arguments are detailed in the Reply Code section.

5.3. Receive

When a client sends a request, the server receives a header with an unsigned char (1 byte), representing the command, and an int for the message size.

You may use the following structure for client replies: 

          
                   
typedef struct __attribute__((packed))
{
    unsigned char command;
    int message_size;
} client_request_t;

The server must wait for the full message to be received before processing the request.

7. Reply Codes

7.1. 2XX

The following reply codes are used for success. They are sent to the user making the request. Here is a list of all of the success reply codes, and their meaning:

200: Command OK.

  • user_id (message_size bytes): a character pointer corresponding to the UUID assigned to the user.

201: when a user asks for another user in particular (/user or /info CLI command).

  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user.
  • user_name: (32 bytes) a character pointer corresponding to the user's name.
  • user_status: (1 bytes) an integer corresponding to the user status (0 = not connected; 1 = connected).

202: when a user asks for the users list (/list CLI command), used in a loop over each team.

  • user_id (16 bytes): a character pointer corresponding to the UUID assigned to the user.
  • user_name (32 bytes): a character pointer corresponding to the user's name.
  • user_status (1 bytes): an integer corresponding to the user status (0 = not connected; 1 = connected).

210: when a user asks for a team in particular (/info CLI command).

  • team_id (16 bytes): a character pointer corresponding to the UUID assigned to the team.
  • team_name (32 bytes): a character pointer corresponding to the team's name.
  • team_description (255 bytes): a character pointer corresponding to the team's description.

211: when a user asks for a channel in particular (/info CLI command).

  • channel_id (16 bytes): a character pointer corresponding to the UUID assigned to the channel.
  • channel_name (32 bytes): a character pointer corresponding to the channel's name.
  • channel_description (255 bytes): a character pointer corresponding to the channel's description.

212: when a user asks for a thread in particular (/info CLI command).

  • thread_id (16 bytes): a character pointer corresponding to the UUID assigned to the thread.
  • user_id (16 bytes): a character pointer corresponding to the UUID assigned to the user who created the thread.
  • thread_timestamp (16 bytes): an unsigned integer corresponding to the timestamp of the thread's creation.
  • thread_title (32 bytes): a character pointer corresponding to the thread title.
  • thread_body (512 bytes): a character pointer corresponding to the thread body.

220: when a user asks for a specific private discussion (/messages CLI command), used in a loop over each message

  • sender_id (16 bytes): a character pointer corresponding to the UUID assigned to the user sending the message
  • timestamp (16 bytes): an unsigned integer corresponding to the timestamp of the message.
  • message (512 bytes): a character pointer corresponding to the message.

230: when a user asks for a team in particular (/info CLI command)

  • team_id: (16 bytes) a character pointer corresponding to the UUID assigned to the team.
  • team_name: (32 bytes) a character pointer corresponding to the team's name.
  • team_description: (255 bytes) a character pointer corresponding to the team's description.

231: when the user select a specific team to be in (/list CLI command), used in a loop over each channel

  • channel_id: (16 bytes) a character pointer corresponding to the UUID assigned to the channel in the team.
  • channel_name: (32 bytes) a character pointer corresponding to the channel's name.
  • channel_description: (255 bytes) a character pointer corresponding to the channel's description.

232: when the user select a specific channel to be in (/list CLI command), used in a loop over each thread

  • thread_id: (16 bytes) a character pointer corresponding to the UUID assigned to the thread in the team.
  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user who created the thread.
  • thread_timestamp: (16 bytes) an unsigned integer corresponding to the timestamp of the thread's creation.
  • thread_title: (32 bytes) a character pointer corresponding to the thread title.
  • thread_body: (512 bytes) a character pointer corresponding to the thread body.

233: when a user asks for a specific thread (/list CLI command), used in a loop over each reply

  • thread_id: (16 bytes) a character pointer corresponding to the UUID assigned to the thread in the channel.
  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user who created the thread.
  • reply_timestamp: (16 bytes) an unsigned integer corresponding to the timestamp of the reply.
  • reply_body: (512 bytes) a character pointer corresponding to the reply.

240: when the user create a reply (/subscribe CLI command)

  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user who subscribed;
  • team_id: (16 bytes) a character pointer corresponding to the UUID assigned to the team the user subscribed to.

241: when the user create a reply (/unsubscribe CLI command)

  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user who unsubscribed;
  • team_id: (16 bytes) a character pointer corresponding to the UUID assigned to the team the user unsubscribed from.

250: when the user create a team (/create CLI command)

  • team_id: (16 bytes) a character pointer corresponding to the UUID assigned to the team.
  • team_name: (32 bytes) a character pointer corresponding to the team's name.
  • team_description: (255 bytes) a character pointer corresponding to the team's description.

251: when the user create a channel (/create CLI command)

  • channel_id: (16 bytes) a character pointer corresponding to the UUID assigned to the channel.
  • channel_name: (32 bytes) a character pointer corresponding to the channel's name.
  • channel_description: (255 bytes) a character pointer corresponding to the channel's description.

252: when the user create a thread (/create CLI command)

  • thread_id: (16 bytes) a character pointer corresponding to the UUID assigned to the thread.
  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user who created the thread.
  • thread_timestamp: (16 bytes) an unsigned integer corresponding to the timestamp of the thread's creation.
  • thread_title: (32 bytes) a character pointer corresponding to the thread title.
  • thread_body: (512 bytes) a character pointer corresponding to the thread body.

253: when the user create a reply (/create CLI command)

  • thread_id: (16 bytes) a character pointer corresponding to the UUID assigned to the thread in the channel.
  • user_id: (16 bytes) a character pointer corresponding to the UUID assigned to the user who created the thread.
  • reply_timestamp: (16 bytes) an unsigned integer corresponding to the timestamp of the reply.
  • reply_body: (512 bytes) a character pointer corresponding to the reply.

7.2. 3XX

Here is the list of reply codes and their arguments used for broadcasting other users:

311: A user has created a team

  • The uuid of the team: 16 bytes
  • The name of the team: 32 bytes
  • The description of the team: 255 bytes

312: A user has created a channel

  • The uuid of the channel: 16 bytes
  • The name of the channel: 32 bytes
  • The description of the channel: 255 bytes

313: A user has created a thread

  • The uuid of the thread: 16 bytes
  • The uuid of the user who created the team: 16 bytes
  • The time since the Epoch when the thread has been created: 16 bytes
  • The title of the thread: 32 bytes
  • The message body of the thread: 512 bytes

320: A user has created a message in a thread

  • The uuid of the team: 16 bytes
  • The uuid of the thread: 16 bytes
  • The uuid of the user: 16 bytes
  • The body of the message: 512 bytes

321: A user has sended a private message

  • The uuid of the user: 16 bytes
  • The body of the message: 512 bytes

330: A user logged in

  • The uuid of the user: 16 bytes
  • The name of the user: 32 bytes

331: A user logged out

  • The uuid of the user: 16 bytes
  • The name of the user: 32 bytes

7.3. 5XX

The following reply codes are used for errors. They are sent to the user making the request. Here is a list of all of the error reply codes, and their meaning:

500: Syntax Error.

  • The error message: string

501: Error in arguments.

  • The error message: string

502: Command not implemented.

  • The error message: string

503: Bad sequence of commands.

  • The error message: string

504: Command not implemented for that parameter.

  • The error message: string

530: Not logged in.

  • The error message: string

540: Already exists.

  • Nothing should be sent after the header

550: Unknown command.

  • The error message: string

551: Unknown team.

  • Unknown team's UUID: 16 bytes

552: Unknown channel.

  • Unknown channel's UUID: 16 bytes

553: Unknown thread.

  • Unknown thread's UUID: 16 bytes

554: Unknown user.

  • Unknown user's UUID: 16 bytes

560: Forbidden, access denied.

  • Unknown user's UUID: 16 bytes
